# บทที่ 5
# สรุปผลและข้อเสนอแนะ

บทนี้เป็นการสรุปผลจากการออกแบบ พัฒนา และทดสอบระบบช่วยค้นหาบุคคลจากภาพกล้องวงจรปิด โดยระบบที่พัฒนาขึ้นมีเป้าหมายหลักเพื่อช่วยลดภาระของผู้ปฏิบัติงานในการค้นหาบุคคลจากวิดีโอจำนวนมาก จากเดิมที่ต้องเปิดดูฟุตเทจด้วยตนเองทีละช่วงเวลา ให้เปลี่ยนเป็นการใช้ระบบคอมพิวเตอร์ช่วยตรวจจับบุคคล วิเคราะห์ลักษณะเสื้อผ้า วิเคราะห์สี จัดเก็บข้อมูล และค้นหาย้อนหลังผ่านเว็บแอปพลิเคชัน

ระบบนี้ไม่ได้มีจุดมุ่งหมายเพื่อยืนยันตัวตนของบุคคลอย่างเด็ดขาด แต่ทำหน้าที่เป็นระบบช่วยคัดกรองข้อมูลจากภาพกล้องวงจรปิด โดยใช้ลักษณะภายนอกที่มองเห็นได้จากภาพ เช่น ประเภทเสื้อผ้า สีเสื้อผ้า กล้องที่พบ ช่วงเวลา และข้อมูล feature embedding ที่ได้จากภาพ เพื่อช่วยให้ผู้ใช้พบผลลัพธ์ที่เกี่ยวข้องได้รวดเร็วขึ้น ผลการดำเนินงานในบทนี้จึงสรุปบนฐานของระบบต้นแบบและผลการทดสอบเชิง flow/contract ตามที่นำเสนอในบทที่ 4 โดยแยกให้ชัดเจนว่าส่วนใดเป็นผลที่ทดสอบแล้ว และส่วนใดควรเพิ่มเติมเมื่อมีการทดสอบกับข้อมูลจริงมากขึ้น

## 5.1 สรุปผลการดำเนินงาน

จากการดำเนินโครงงาน สามารถพัฒนาระบบต้นแบบสำหรับช่วยค้นหาบุคคลจากภาพกล้องวงจรปิดได้ครบตาม flow หลักที่วางไว้ ระบบประกอบด้วยส่วนรับข้อมูลวิดีโอหรือสตรีมกล้อง ส่วนประมวลผลภาพด้วยโมเดล AI ส่วนติดตามบุคคล ส่วนวิเคราะห์เสื้อผ้าและสี ส่วนจัดเก็บข้อมูลในฐานข้อมูลและ object storage ส่วน API และส่วนติดต่อผู้ใช้ผ่านเว็บแอปพลิเคชัน

กระบวนการประมวลผลหลักของระบบใช้แนวคิดการ predict 2 รอบ โดยรอบแรกตรวจจับบุคคลและสร้างข้อมูลที่ใช้สำหรับการติดตาม ส่วนรอบที่สองตรวจจับลักษณะเสื้อผ้าบนพื้นที่ของบุคคลที่พบ จากนั้นนำผลการตรวจจับเสื้อผ้าไปวิเคราะห์สีและจัดกลุ่มสี เพื่อให้ข้อมูลที่ได้สามารถนำไปใช้ค้นหาย้อนหลังได้ ระบบยังมีการสร้าง feature embedding ขนาด 768 มิติ เพื่อช่วยเปรียบเทียบความคล้ายของบุคคลในกรณีที่ track id มีโอกาสหลุดหรือเกิดความไม่ต่อเนื่อง

ในด้านการติดตามบุคคล ระบบใช้ ByteTrack ร่วมกับ YOLO11 เพื่อให้ได้ตำแหน่งบุคคลและ track id ในแต่ละเฟรม โดยมีแนวทางตรวจสอบตำแหน่งเดิมของบุคคลและใช้ข้อมูลสีร่วมกับ feature vector เพื่อช่วยพิจารณากรณีที่ track id เปลี่ยน ระบบยังไม่ได้ทำ re-identification ข้ามกล้องแบบสมบูรณ์ แต่โครงสร้างข้อมูลและแนวคิดของระบบถูกออกแบบให้สามารถต่อยอดไปสู่การเชื่อมโยงกล้องหลายตัวในอนาคตได้

ในด้านการจัดเก็บข้อมูล ระบบมีการเตรียมข้อมูลสำหรับบันทึกลง PostgreSQL เช่น `camera_id`, `video_id`, `track_id`, `bbox`, `class_name`, `category`, `image_path`, `video_time_offset` และ `embedding` ส่วนภาพบุคคลที่ถูก crop จากผลตรวจจับสามารถจัดเก็บใน MinIO เพื่อใช้เป็นหลักฐานประกอบผลการค้นหาและแสดงในหน้าเว็บ ระบบจึงสามารถแยกการเก็บ metadata ออกจากการเก็บไฟล์ภาพ ซึ่งเหมาะกับงานที่ต้องจัดการข้อมูลวิดีโอและภาพจำนวนมาก

ในด้าน frontend ระบบใช้ Next.js และ TypeScript เพื่อสร้างหน้าจอสำหรับใช้งาน เช่น dashboard, realtime processing, camera management และ search/investigation ผู้ใช้สามารถค้นหาด้วยเงื่อนไข เช่น ประเภทเสื้อผ้า สี กล้อง วิดีโอ และช่วงเวลา รวมถึงมีแนวคิดรองรับการค้นหาด้วยรูปภาพ input เพื่อให้ระบบดึงคุณลักษณะจากภาพแล้วนำไปค้นหาในฐานข้อมูล

## 5.2 สรุปผลตามวัตถุประสงค์ของโครงงาน

เมื่อพิจารณาจากวัตถุประสงค์ของโครงงาน ระบบสามารถบรรลุเป้าหมายหลักในระดับระบบต้นแบบได้ โดยสามารถประมวลผลภาพจากวิดีโอ ตรวจจับบุคคล วิเคราะห์ลักษณะเสื้อผ้า วิเคราะห์สี จัดเก็บข้อมูล และค้นหาย้อนหลังได้ อย่างไรก็ตาม ผลลัพธ์ด้านความแม่นยำของโมเดลจริงและความเร็วในสภาพแวดล้อมใช้งานจริงยังควรมีการทดสอบเพิ่มเติมด้วยชุดข้อมูลจริงที่มี ground truth

ตารางที่ 5.2 แสดงการประเมินผลตามวัตถุประสงค์ของโครงงาน

| วัตถุประสงค์ | ผลการดำเนินงาน | ข้อสรุป |
|---|---|---|
| พัฒนาระบบช่วยค้นหาบุคคลจากภาพกล้องวงจรปิด | ระบบสามารถรับวิดีโอ ประมวลผลบุคคล และสร้างข้อมูลสำหรับค้นหาย้อนหลังได้ | บรรลุในระดับต้นแบบ |
| ลดภาระการค้นหาฟุตเทจด้วยคนเพียงอย่างเดียว | ระบบช่วยแปลงวิดีโอเป็นข้อมูลที่ค้นหาได้ตามกล้อง เวลา เสื้อผ้า และสี | บรรลุในเชิงแนวทางและ flow |

| จัดเก็บข้อมูลสำหรับค้นหาย้อนหลัง | เตรียม payload สำหรับ PostgreSQL และจัดเก็บภาพใน MinIO | บรรลุในระดับ contract และระบบต้นแบบ |

จากตารางข้างต้นสามารถสรุปได้ว่าโครงงานนี้บรรลุวัตถุประสงค์หลักของการพัฒนาระบบช่วยค้นหาบุคคลจากกล้องวงจรปิดในระดับต้นแบบ แต่ยังไม่ควรสรุปว่าเป็นระบบ production ที่พร้อมใช้งานจริงทุกสถานการณ์ เนื่องจากยังต้องมีการทดสอบกับวิดีโอจริงจำนวนมาก การประเมินความแม่นยำของโมเดล และการทดสอบ performance บน hardware เป้าหมายเพิ่มเติม

## 5.3 สรุปผลการทดสอบระบบ

ผลการทดสอบในบทที่ 4 ใช้ชุดทดสอบที่สร้างขึ้นใหม่สำหรับตรวจสอบ flow การทำงานปัจจุบันของระบบ โดยไม่อ้างอิงผลทดสอบเดิม ชุดทดสอบอยู่ในไฟล์ `tests/chapter4/test_chapter4_flow_data.py` และรันด้วยคำสั่ง `uv run python -m pytest tests/chapter4/test_chapter4_flow_data.py -v` ผลการทดสอบผ่านทั้งหมด 5 รายการจาก 5 รายการ

การทดสอบใช้ synthetic video และ fake AI component เพื่อให้ผลลัพธ์คงที่และตรวจสอบได้ซ้ำ จุดประสงค์ของการทดสอบนี้ไม่ใช่การวัด accuracy ของโมเดลจริง แต่เป็นการตรวจสอบว่าแต่ละส่วนของระบบส่งต่อข้อมูลได้ถูกต้อง เช่น FrameProcessor ส่งผลลัพธ์การตรวจจับตาม contract, VideoProcessor นับเฟรมและ detection ได้ถูกต้อง, payload สำหรับฐานข้อมูลมี field สำคัญครบ และ advanced search สร้าง query ได้ตรงตามเงื่อนไข

ตารางที่ 5.3 สรุปผลการทดสอบหลักของบทที่ 4

## 5.4 ปัญหาและอุปสรรคที่พบ

ระหว่างการพัฒนาและทดสอบระบบ พบปัญหาและข้อจำกัดหลายด้าน ทั้งด้านคุณภาพภาพ ด้านโมเดล AI ด้าน hardware ด้านฐานข้อมูล และด้านกฎหมายคุ้มครองข้อมูลส่วนบุคคล ปัญหาเหล่านี้เป็นสิ่งสำคัญที่ต้องพิจารณาก่อนนำระบบไปใช้กับสถานการณ์จริง

### 5.4.1 ปัญหาด้านคุณภาพภาพจากกล้องวงจรปิด

กล้องวงจรปิดในสถานการณ์จริงมักมีข้อจำกัดด้านความละเอียด มุมกล้อง แสง เงา ระยะห่างของบุคคลจากกล้อง การบังกันของคนหลายคน และ motion blur ปัจจัยเหล่านี้ส่งผลโดยตรงต่อการตรวจจับบุคคล การจำแนกเสื้อผ้า และการวิเคราะห์สี ระบบปัจจุบันพยายามลดผลกระทบจากแสงบางส่วนด้วยแนวทางการจัดกลุ่มสี แต่หากปัญหาของภาพรุนแรงมาก เช่น ภาพมืดเกินไป เงาบังเต็มตัว หรือความละเอียดต่ำจนไม่เห็นรายละเอียดเสื้อผ้า ระบบยังไม่สามารถแก้ปัญหาได้ทั้งหมด

### 5.4.2 ปัญหาด้าน tracking และ re-identification

ระบบใช้ ByteTrack เพื่อช่วยติดตามบุคคลภายในวิดีโอ แต่ในสถานการณ์ที่คนเดินผ่านกัน บังกัน หายออกจากเฟรม หรือกลับเข้ามาใหม่ อาจเกิดปัญหา track id เปลี่ยนหรือหลุดได้ ระบบจึงใช้ข้อมูลตำแหน่ง สี และ feature vector เพื่อช่วยเปรียบเทียบในบางกรณี อย่างไรก็ตาม การเปรียบเทียบที่ใช้ในปัจจุบันยังไม่ใช่ระบบ Re-ID ที่ซับซ้อนเต็มรูปแบบ และยังไม่ได้รองรับการ track ข้ามกล้องแบบสมบูรณ์

### 5.4.3 ปัญหาด้านข้อมูลและการ train โมเดล

ข้อมูลที่ใช้พัฒนาโมเดลมาจาก PA-100K และ DeepFashion2 รวมถึงการ label เพิ่มด้วยตนเองประมาณ 4,000 รูปต่อชุดข้อมูล ซึ่งช่วยให้ระบบสามารถเรียนรู้ลักษณะเสื้อผ้าได้ในระดับหนึ่ง แต่ข้อมูลดังกล่าวอาจยังไม่ครอบคลุมสภาพแวดล้อมจริงของกล้องวงจรปิด เช่น ภาพจากมุมสูง ภาพกลางคืน ภาพจากกล้องคุณภาพต่ำ หรือภาพที่มีคนจำนวนมากในเฟรม นอกจากนี้ หากใช้ภาพบุคคลจากอินเทอร์เน็ต ต้องพิจารณาเรื่องสิทธิ์การใช้ข้อมูลและข้อกำหนดด้านข้อมูลส่วนบุคคลเพิ่มเติม

### 5.4.4 ปัญหาด้าน hardware และ performance

เครื่องที่ใช้พัฒนามีข้อจำกัดด้านทรัพยากร ได้แก่ GPU NVIDIA GTX 1650 VRAM 4GB และ RAM 16GB ซึ่งส่งผลต่อการเลือกขนาดโมเดล ความละเอียดภาพที่ประมวลผลได้ ค่า frame skip และความเร็วโดยรวมของระบบ ข้อจำกัดนี้ทำให้ระบบต้องเน้นการประมวลผลแบบประหยัดทรัพยากร เช่น การไม่ประมวลผลทุกเฟรม และการบันทึกภาพบางช่วงแทนการบันทึกทุกเฟรม

### 5.4.5 ปัญหาด้านฐานข้อมูลและการจัดเก็บ embedding

ระบบมีแนวคิดจัดเก็บ embedding เพื่อช่วยค้นหาหรือเปรียบเทียบความคล้ายของบุคคล แต่ embedding ปัจจุบันมีขนาด 768 มิติ จึงต้องพิจารณาวิธีจัดเก็บให้เหมาะสม หากเก็บเป็น JSON หรือ array ทั่วไปใน PostgreSQL อาจเพียงพอสำหรับการบันทึกและอ่านข้อมูลในระดับต้นแบบ แต่เมื่อข้อมูลมีจำนวนมาก การค้นหา similarity อาจช้าลงและจัด index ได้ยาก ดังนั้นหากต้องใช้ embedding เป็นส่วนสำคัญของการค้นหา ควรพิจารณาใช้ `pgvector` หรือ vector database เพื่อรองรับการค้นหาเชิงความคล้ายอย่างมีประสิทธิภาพ

### 5.4.6 ปัญหาด้านกฎหมายและความเป็นส่วนตัว

ระบบนี้เกี่ยวข้องกับภาพบุคคล ซึ่งเข้าข่ายข้อมูลส่วนบุคคลและอาจเป็นข้อมูลที่มีความอ่อนไหวต่อการนำไปใช้ผิดวัตถุประสงค์ แม้ระบบจะไม่ได้ใช้การจดจำใบหน้า แต่การเก็บภาพบุคคล ตำแหน่ง เวลา และข้อมูลลักษณะภายนอกยังต้องมีมาตรการควบคุมการเข้าถึง เช่น role-based access control, audit log, การกำหนดระยะเวลาการเก็บข้อมูล และการจำกัดสิทธิ์เฉพาะผู้เกี่ยวข้อง

ตารางที่ 5.5 สรุปปัญหาและแนวทางรับมือ

| ปัญหา | ผลกระทบ | แนวทางรับมือ |
|---|---|---|
| แสง เงา และมุมกล้องไม่เหมาะสม | ตรวจจับผิด วิเคราะห์สีผิด หรือไม่พบเสื้อผ้า | ใช้กล้องที่มีคุณภาพเหมาะสม เพิ่มข้อมูล train และปรับ color grouping |
| Track id หลุด | คนเดียวกันอาจถูกนับเป็นหลาย id | ใช้ตำแหน่ง สี และ embedding ช่วยเปรียบเทียบ และพัฒนา Re-ID เพิ่ม |
| ข้อมูล train ยังไม่ครอบคลุม | ความแม่นยำลดลงเมื่อเจอสภาพแวดล้อมใหม่ | เพิ่ม dataset จากกล้องจริงและทำ evaluation แยกตามสภาพแวดล้อม |
| GPU และ RAM จำกัด | ประมวลผลช้า ต้องลด frame rate หรือความละเอียด | ใช้ frame skip, model ขนาดเล็ก, batch optimization หรือ TensorRT |
| การเก็บ embedding จำนวนมาก | ฐานข้อมูลอาจช้าเมื่อค้นหา similarity | ใช้ pgvector หรือ vector database และออกแบบ index |
| ข้อมูลภาพบุคคล | เสี่ยงต่อ PDPA และการใช้งานผิดวัตถุประสงค์ | จำกัดสิทธิ์ผู้ใช้ ทำ audit log และกำหนด data retention |

## 5.5 ข้อจำกัดของระบบปัจจุบัน

ระบบที่พัฒนาขึ้นเป็นระบบต้นแบบสำหรับช่วยค้นหาบุคคล ไม่ใช่ระบบพิสูจน์ตัวตนหรือระบบยืนยันบุคคลแบบสมบูรณ์ ผลลัพธ์ที่ระบบแสดงควรถูกใช้เป็นข้อมูลช่วยค้นหาและช่วยคัดกรอง โดยผู้ใช้งานยังควรตรวจสอบผลลัพธ์ด้วยตนเองก่อนนำไปใช้ประกอบการตัดสินใจ

ข้อจำกัดที่สำคัญของระบบปัจจุบันประกอบด้วย ระบบยังไม่รองรับการติดตามข้ามกล้องแบบเต็มรูปแบบ การเปรียบเทียบ feature vector ยังอยู่ในระดับช่วยประกอบการตัดสินใจ ไม่ใช่โมเดล Re-ID ที่ train เฉพาะทางเต็มรูปแบบ การวิเคราะห์สีอาจผิดพลาดเมื่อภาพมีแสงหรือเงารุนแรง และการค้นหาขึ้นอยู่กับคุณภาพของ metadata ที่ตรวจจับได้ หากขั้นตอนตรวจจับหรือจำแนกเสื้อผ้าผิด ผลการค้นหาก็มีโอกาสผิดหรือขาดข้อมูลตามไปด้วย

นอกจากนี้ ผลทดสอบล่าสุดในบทที่ 4 เป็นการทดสอบเชิง deterministic โดยใช้ข้อมูลจำลองและ fake AI component จึงเหมาะสำหรับยืนยัน flow และ data contract แต่ยังไม่สามารถใช้สรุปความแม่นยำของโมเดลจริงหรือความเร็วจริงของระบบได้ การนำระบบไปใช้งานจริงควรต้องมีการทดสอบเพิ่มเติมกับวิดีโอกล้องวงจรปิดจริงและข้อมูล ground truth

ตารางที่ 5.6 แสดงข้อจำกัดของระบบและความหมายต่อการใช้งาน

| ข้อจำกัด | ความหมายต่อการใช้งาน |
|---|---|
| ไม่ใช่ระบบยืนยันตัวตน | ผลลัพธ์ควรใช้เป็นข้อมูลช่วยค้นหา ไม่ใช่หลักฐานยืนยันบุคคลโดยตรง |
| ยังไม่มี cross-camera Re-ID สมบูรณ์ | การติดตามบุคคลระหว่างกล้องหลายตัวต้องพัฒนาต่อ |
| ยังไม่ได้วัด accuracy จากข้อมูลจริง | ยังไม่ควรสรุปตัวเลข precision, recall หรือ mAP |
| ยังไม่ได้วัด performance ด้วยโมเดลจริงครบชุด | ค่า effective FPS จาก test ปัจจุบันไม่ใช่ค่าใช้งานจริง |
| คุณภาพภาพส่งผลสูง | กล้องที่แสงน้อย มุมไม่ดี หรือความละเอียดต่ำจะทำให้ผลลัพธ์ลดลง |
| การจัดเก็บข้อมูลบุคคลต้องระวัง | ต้องมีนโยบายสิทธิ์ การเก็บรักษา และการลบข้อมูล |

## 5.6 ข้อเสนอแนะในการนำระบบไปใช้งาน

หากนำระบบไปใช้งานจริง ควรใช้ระบบในฐานะเครื่องมือช่วยผู้ปฏิบัติงาน ไม่ใช่เครื่องมือที่ตัดสินผลแทนมนุษย์ทั้งหมด ผู้ใช้ควรตรวจสอบภาพต้นฉบับและบริบทของวิดีโอร่วมกับผลการค้นหาทุกครั้ง โดยเฉพาะกรณีที่เกี่ยวข้องกับการระบุตัวบุคคลหรือการใช้ข้อมูลในทางกฎหมาย

ด้านการติดตั้งกล้อง ควรเลือกมุมกล้องที่เห็นลักษณะเสื้อผ้าชัดเจน หลีกเลี่ยงมุมที่บุคคลเล็กเกินไปในภาพ และควรมีแสงที่เพียงพอ เพราะคุณภาพของภาพเป็นปัจจัยหลักต่อผลลัพธ์ของระบบ AI หากกล้องมีความละเอียดต่ำมากหรืออยู่ในตำแหน่งที่เห็นคนเพียงบางส่วน ระบบอาจไม่สามารถวิเคราะห์เสื้อผ้าหรือสีได้แม่นยำ

ด้านการจัดเก็บข้อมูล ควรกำหนดนโยบายการเก็บรักษาภาพและ metadata อย่างชัดเจน เช่น เก็บเฉพาะระยะเวลาที่จำเป็น ลบข้อมูลเมื่อพ้นระยะเวลาการใช้งาน จำกัดสิทธิ์การเข้าถึงเฉพาะผู้เกี่ยวข้อง และบันทึกประวัติการเข้าถึงข้อมูลทุกครั้ง เพื่อลดความเสี่ยงด้าน PDPA และการใช้งานผิดวัตถุประสงค์

ด้านการทดสอบระบบ ควรจัดทำชุดทดสอบกับวิดีโอจริงในหลายสภาพแวดล้อม เช่น กลางวัน กลางคืน พื้นที่แสงน้อย พื้นที่คนหนาแน่น และกล้องหลายมุม พร้อมจัดทำ ground truth เพื่อประเมิน precision, recall, F1-score, mAP และ confusion matrix ของโมเดลตรวจจับเสื้อผ้าและสี

ตารางที่ 5.7 แสดงข้อเสนอแนะสำหรับการใช้งานจริง

| ด้าน | ข้อเสนอแนะ |
|---|---|
| การใช้งานโดยผู้ใช้ | ใช้ระบบเป็นตัวช่วยคัดกรอง และให้ผู้ใช้ตรวจสอบผลลัพธ์ด้วยภาพจริง |
| ตำแหน่งกล้อง | เลือกมุมที่เห็นตัวบุคคลและเสื้อผ้าชัดเจน |
| แสงและสภาพแวดล้อม | ลดเงารุนแรงและพื้นที่ย้อนแสงเท่าที่ทำได้ |
| การตั้งค่า frame skip | ปรับให้เหมาะกับ hardware และความต้องการด้านความละเอียดเวลา |
| การจัดเก็บข้อมูล | กำหนด data retention และควบคุมสิทธิ์การเข้าถึง |
| การตรวจสอบย้อนหลัง | เก็บ log การค้นหาและการเข้าถึงข้อมูล |
| การอัปเดตโมเดล | ทดสอบโมเดลกับข้อมูลจริงก่อนเปลี่ยนเวอร์ชันในระบบ |

## 5.7 แนวทางพัฒนาต่อในอนาคต

ระบบนี้ยังสามารถพัฒนาต่อยอดได้หลายด้าน โดยเฉพาะการเพิ่มความแม่นยำของโมเดล การเพิ่มประสิทธิภาพการประมวลผล การค้นหาด้วย embedding และการเชื่อมโยงข้อมูลระหว่างกล้องหลายตัว

แนวทางแรกคือการทดสอบและปรับปรุงโมเดลด้วยชุดข้อมูลจริง ควรเก็บวิดีโอจากสถานการณ์ใช้งานจริงและทำ ground truth สำหรับบุคคล เสื้อผ้า สี และ track id เพื่อใช้วัดผลอย่างเป็นระบบ จากนั้นนำผลที่ได้มาปรับปรุง dataset, train โมเดลเพิ่มเติม และวิเคราะห์ข้อผิดพลาดเป็นราย class เช่น long_sleeve, short_sleeve, dress, skirt, shorts และ trousers

แนวทางที่สองคือการปรับปรุงระบบ Re-ID และ similarity matching ปัจจุบันระบบใช้สีและ feature vector เพื่อช่วยเปรียบเทียบในกรณี track id มีปัญหา แต่ในอนาคตควรพัฒนาให้รองรับการค้นหาบุคคลที่คล้ายกันจาก embedding ได้มากขึ้น โดยใช้ `pgvector` หรือ vector database เพื่อรองรับ approximate nearest neighbor search และกำหนด threshold จากผลทดสอบจริง

แนวทางที่สามคือการพัฒนาความสัมพันธ์ระหว่างกล้อง ระบบในอนาคตอาจเก็บข้อมูลตำแหน่งกล้อง เส้นทางเดิน และเวลาที่คาดว่าบุคคลจะเดินจากกล้องหนึ่งไปยังอีกกล้องหนึ่ง เพื่อช่วยเสนอรายชื่อกล้องที่ควรค้นหาต่อโดยอัตโนมัติ เมื่อบุคคลหายไปจากกล้องหนึ่ง ระบบอาจเตรียม query สำหรับกล้องใกล้เคียงและใช้คุณลักษณะเดิม เช่น สี เสื้อผ้า และ embedding เพื่อค้นหาต่อ

แนวทางที่สี่คือการเพิ่มความปลอดภัยของระบบ ควรพัฒนา role-based access control ให้ชัดเจน เพิ่ม audit log สำหรับการเข้าดูภาพและการค้นหา เพิ่มหน้าจอจัดการสิทธิ์ผู้ใช้ และกำหนดนโยบายการลบข้อมูลอัตโนมัติเมื่อพ้นระยะเวลาที่กำหนด

แนวทางที่ห้าคือการเพิ่มชุดทดสอบแบบ integration และ end-to-end โดยทดสอบการทำงานร่วมกันของ video processing, PostgreSQL, MinIO, API และ frontend เพื่อให้มั่นใจว่าระบบทำงานได้ครบตั้งแต่รับวิดีโอจนถึงแสดงผลการค้นหาในหน้าเว็บ

ตารางที่ 5.8 แสดงแผนพัฒนาต่อที่ควรดำเนินการ

| ลำดับ | แนวทางพัฒนา | ความสำคัญ | ผลที่คาดว่าจะได้ |
|---|---|---|---|
| 1 | ทดสอบโมเดลกับวิดีโอจริงและ ground truth | สูง | ได้ตัวเลข accuracy และข้อผิดพลาดจริงของระบบ |
| 2 | เพิ่ม `pgvector` หรือ vector database | สูง | ค้นหาและเปรียบเทียบ embedding ได้เร็วขึ้น |
| 3 | ปรับปรุง Re-ID ในกรณี track id หลุด | สูง | ลดการนับคนซ้ำและเพิ่มความต่อเนื่องของผลติดตาม |
| 4 | ทดสอบ performance ด้วยโมเดลจริง | สูง | ทราบ FPS, latency, RAM และ VRAM จริง |
| 5 | พัฒนาความสัมพันธ์ระหว่างกล้อง | กลาง | รองรับการค้นหาต่อเนื่องระหว่างกล้อง |
| 6 | เพิ่ม RBAC, audit log และ retention policy | สูง | ลดความเสี่ยงด้านข้อมูลส่วนบุคคล |
| 7 | เพิ่ม integration test กับ PostgreSQL และ MinIO | กลาง | ลดความเสี่ยงจากปัญหาระบบเชื่อมต่อกัน |
| 8 | ปรับปรุง UI รายงานผลและการกรองข้อมูล | กลาง | ทำให้ผู้ใช้ตรวจสอบผลลัพธ์ได้สะดวกขึ้น |
| 9 | Optimize inference เช่น batching หรือ TensorRT | กลาง | เพิ่มความเร็วและลดภาระ GPU |

## 5.8 สิ่งที่ควรเพิ่มเติมก่อนจัดทำเล่มสมบูรณ์

แม้เนื้อหาบทที่ 5 สามารถสรุปจากผลการพัฒนาและผลทดสอบล่าสุดได้แล้ว แต่ก่อนจัดทำเล่มสมบูรณ์ควรเพิ่มเติมข้อมูลบางส่วนเพื่อให้รายงานมีความครบถ้วนและน่าเชื่อถือมากขึ้น โดยเฉพาะข้อมูลที่ต้องมาจากการทดลองจริงหรือการตัดสินใจขั้นสุดท้ายของผู้จัดทำ

ตารางที่ 5.9 แสดงรายการข้อมูลที่ควรเพิ่มเติมก่อนส่งเล่ม

| รายการที่ควรเพิ่ม | เหตุผล |
|---|---|
| ชื่อโครงงานอย่างเป็นทางการ | ใช้ให้ตรงกันทุกบท หน้าปก สารบัญ และเอกสารส่งอาจารย์ |
| วัตถุประสงค์จากบทที่ 1 ฉบับสุดท้าย | เพื่อให้ตารางสรุปผลตามวัตถุประสงค์ในบทที่ 5 ตรงกับบทที่ 1 |
| ภาพหน้าจอระบบจริง | ใช้ประกอบบทที่ 4 และช่วยยืนยันว่าระบบมีการพัฒนาจริง |
| ผลทดสอบกับวิดีโอจริง | ใช้สรุป accuracy และ performance จริงของระบบ |
| ตาราง confusion matrix หรือ metric ของโมเดล | ใช้ประเมินความแม่นยำของ clothing detection และ color analysis |
| รายละเอียด hardware ที่ใช้ทดสอบจริง | ใช้ประกอบการวิเคราะห์ performance และข้อจำกัด |
| นโยบาย PDPA ที่เลือกใช้ | ใช้ยืนยันมาตรการลดความเสี่ยงด้านข้อมูลส่วนบุคคล |
| สถานะสุดท้ายของ PostgreSQL, MinIO และ embedding storage | ใช้สรุปความพร้อมของระบบจัดเก็บข้อมูล |

ข้อมูลเหล่านี้ไม่ได้ทำให้บทที่ 5 ปัจจุบันใช้ไม่ได้ แต่เป็นข้อมูลที่ควรเติมภายหลังเพื่อให้รายงานฉบับสมบูรณ์มีน้ำหนักมากขึ้น โดยเฉพาะหากอาจารย์ต้องการเห็นผลทดสอบจากข้อมูลจริงหรือหลักฐานเชิงภาพประกอบการทำงานของระบบ

## 5.9 สรุปบท

จากการพัฒนาและทดสอบระบบ สามารถสรุปได้ว่าโครงงานนี้สามารถสร้างระบบต้นแบบสำหรับช่วยค้นหาบุคคลจากภาพกล้องวงจรปิดได้ตามแนวทางที่กำหนดไว้ ระบบสามารถตรวจจับบุคคล วิเคราะห์เสื้อผ้า วิเคราะห์สี จัดเก็บข้อมูล และรองรับการค้นหาย้อนหลังผ่านเว็บแอปพลิเคชัน โดยผลการทดสอบ flow ล่าสุดผ่านทั้งหมด 5 รายการจาก 5 รายการ แสดงให้เห็นว่าส่วนประกอบหลักของระบบทำงานร่วมกันได้ถูกต้องตาม contract ที่กำหนด

อย่างไรก็ตาม ระบบยังมีข้อจำกัดด้านคุณภาพภาพจากกล้องวงจรปิด ความแม่นยำของโมเดลในสภาพแวดล้อมจริง การติดตามบุคคลเมื่อ track id หลุด การค้นหาข้ามกล้อง และข้อจำกัดด้าน hardware รวมถึงต้องให้ความสำคัญกับการคุ้มครองข้อมูลส่วนบุคคลตามหลัก PDPA ดังนั้นระบบนี้ควรถูกมองเป็นต้นแบบที่มีศักยภาพในการช่วยลดภาระการค้นหาวิดีโอ แต่ยังต้องทดสอบกับข้อมูลจริง ปรับปรุงโมเดล เพิ่มมาตรการความปลอดภัย และพัฒนาประสิทธิภาพก่อนนำไปใช้งานในสภาพแวดล้อมจริงอย่างเต็มรูปแบบ
