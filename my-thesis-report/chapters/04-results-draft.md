# บทที่ 4
# ผลการดำเนินงานและการทดสอบระบบ

บทนี้นำเสนอผลการดำเนินงานและผลการทดสอบระบบช่วยค้นหาบุคคลจากภาพกล้องวงจรปิด โดยเนื้อหาในบทนี้ปรับปรุงใหม่จากผลการทดสอบชุดล่าสุดที่ออกแบบขึ้นสำหรับบทที่ 4 โดยเฉพาะ ไม่อ้างอิงผลการทดสอบเดิม และไม่ใช้ข้อมูลจากรายงานเก่าในการสรุปผล ระบบที่พัฒนาขึ้นมีเป้าหมายเพื่อช่วยลดภาระการค้นหาบุคคลจากวิดีโอกล้องวงจรปิดจำนวนมาก ด้วยการตรวจจับบุคคล ติดตามบุคคล วิเคราะห์ลักษณะเสื้อผ้า วิเคราะห์สี จัดเก็บผลการตรวจจับ และเปิดให้ผู้ใช้ค้นหาย้อนหลังผ่านเว็บแอปพลิเคชัน

แนวทางการนำเสนอในบทนี้แบ่งออกเป็นหลายส่วนตามลักษณะของตัวอย่างบทที่ 4 ที่ใช้เป็นแนวทาง ได้แก่ ผลการพัฒนาระบบ ผลการทดสอบระบบ ผลการวิเคราะห์ผลลัพธ์ตาม flow การทำงาน ผลการประเมินตามวัตถุประสงค์ การเปรียบเทียบกับระบบเดิม ปัญหาที่พบและแนวทางแก้ไข รวมถึงข้อจำกัดของการทดสอบในปัจจุบัน เนื่องจากระบบนี้เกี่ยวข้องกับโมเดล AI และข้อมูลภาพบุคคล จึงแยกให้ชัดเจนระหว่างผลการทดสอบเชิง flow/contract ที่ทำแล้ว และผลการทดสอบเชิงความแม่นยำของโมเดลจริงที่ควรเพิ่มเติมภายหลัง

ผลการทดสอบหลักในบทนี้มาจากไฟล์ `tests/chapter4/test_chapter4_flow_data.py` และถูกรวบรวมเป็นไฟล์ `my-thesis-report/qa/chapter4-flow-test-results.json` ซึ่งสร้างเมื่อวันที่ 13 พฤษภาคม 2569 เวลา 13:26:14 น. ตามเวลา Asia/Bangkok การทดสอบใช้ข้อมูลจำลองแบบ deterministic เพื่อให้ตรวจสอบ flow ของระบบได้ซ้ำและควบคุมผลลัพธ์ได้แน่นอน โดยยังไม่ได้ใช้วิดีโอจริงหรือโมเดลจริงในการประเมิน accuracy ขั้นสุดท้าย

## 4.1 ผลการพัฒนาระบบ

ผลจากการพัฒนาโครงงานทำให้ได้ระบบต้นแบบสำหรับช่วยค้นหาบุคคลจากภาพกล้องวงจรปิด ซึ่งประกอบด้วยส่วนประมวลผลภาพและวิดีโอ ส่วนจัดเก็บข้อมูล ส่วน API และส่วนติดต่อผู้ใช้ ระบบสามารถรับข้อมูลจากวิดีโอหรือสตรีมกล้อง นำเฟรมเข้าสู่กระบวนการตรวจจับบุคคล วิเคราะห์เสื้อผ้าและสี จากนั้นบันทึกผลลัพธ์ในรูปแบบที่ค้นหาได้ผ่านเว็บแอปพลิเคชัน

ระบบถูกออกแบบให้รองรับการใช้งานโดยผู้ใช้ที่ต้องการค้นหาบุคคลจากลักษณะภายนอก เช่น ประเภทเสื้อผ้า สี กล้อง วิดีโอ หรือช่วงเวลา แทนการเปิดดูวิดีโอทั้งหมดด้วยตนเอง จุดเด่นของระบบคือไม่ได้มุ่งให้ AI ตัดสินแทนมนุษย์ทั้งหมด แต่ทำหน้าที่เป็นเครื่องมือช่วยคัดกรองและจัดระเบียบข้อมูล เพื่อให้ผู้ปฏิบัติงานใช้เวลาน้อยลงในการค้นหาข้อมูลที่เกี่ยวข้อง

### 4.1.3 ผลลัพธ์ด้านโครงสร้างข้อมูล

ระบบมีการจัดเก็บข้อมูลการตรวจจับในลักษณะที่รองรับการค้นหาย้อนหลัง โดยข้อมูลสำคัญประกอบด้วยรหัสกล้อง รหัสวิดีโอ track id ตำแหน่งของบุคคลในเฟรม ประเภทเสื้อผ้า สี ภาพที่จัดเก็บ และ feature embedding ข้อมูลเหล่านี้ทำให้สามารถนำผลการตรวจจับไปใช้ในหลายมิติ เช่น การค้นหาตามสี การค้นหาตามชนิดเสื้อผ้า การดูผลตามกล้อง หรือการวิเคราะห์ความต่อเนื่องของบุคคลในวิดีโอ

ตารางที่ 4.3 แสดงข้อมูลหลักที่ระบบสร้างจากการตรวจจับ

| ข้อมูล | ตัวอย่าง | ประโยชน์ |

| `camera_id` | `CH4-CAM-01` | ระบุแหล่งที่มาของวิดีโอหรือกล้อง |

| `class_name` | `long_sleeve`, `trousers` | ใช้ค้นหาตามประเภทเสื้อผ้า |

## 4.2 แนวทางการทดสอบระบบ

### 4.3.1 การทดสอบ FrameProcessor

การทดสอบ `FrameProcessor` มีเป้าหมายเพื่อตรวจสอบว่ากระบวนการประมวลผลภาพหนึ่งเฟรมทำงานตามแนวคิด two-pass detection ที่อธิบายในบทที่ 3 หรือไม่ โดยจำลองให้ detector ตรวจพบคน 1 คน จากนั้น classifier คืนผลเสื้อผ้า 3 อันดับแรก ได้แก่ `long_sleeve`, `trousers` และ `shorts` แล้วให้ระบบเลือกผลลัพธ์ตามกฎที่กำหนด คือเลือกเสื้อผ้าส่วนบนได้ไม่เกิน 1 รายการ และเสื้อผ้าส่วนล่างได้ไม่เกิน 1 รายการ

ผลการทดสอบพบว่า `FrameProcessor` สามารถสร้างผลลัพธ์สถานะ `success` ตรวจพบบุคคล 1 คน เลือกเสื้อผ้า 2 รายการ ได้แก่ `long_sleeve` และ `trousers` พร้อมสร้าง embedding ขนาด 768 มิติ และจัดกลุ่มสีเป็น `blue_tones` และ `dark_colors`

| Metric | ผลลัพธ์ |
|---|---|
| สถานะการประมวลผล | success |
| จำนวนบุคคลที่ตรวจพบ | 1 |
| จำนวนเสื้อผ้าที่เลือกต่อบุคคล | 2 |
| Class เสื้อผ้าที่เลือก | `long_sleeve`, `trousers` |
| กลุ่มสีที่ได้ | `blue_tones`, `dark_colors` |
| ขนาด embedding | 768 มิติ |

ผลลัพธ์นี้แสดงให้เห็นว่า logic หลักของการประมวลผลภาพหนึ่งเฟรมสามารถส่งต่อข้อมูลสำคัญได้ครบ ได้แก่ bounding box ของคน class เสื้อผ้า สี และ feature vector อย่างไรก็ตามผลนี้ยังไม่ใช่ค่าความแม่นยำของโมเดลจริง เพราะใช้ component จำลองเพื่อควบคุมผลลัพธ์

### 4.3.2 การทดสอบกฎการเลือกเสื้อผ้า

ระบบจำแนกเสื้อผ้าอาจได้รับผลทำนายหลาย class จาก classifier แต่ไม่ได้บันทึกทุก class โดยตรง เพราะการบันทึกทุกผลทำนายอาจทำให้ข้อมูลซ้ำและค้นหายาก ระบบจึงใช้กฎการเลือกผลลัพธ์เพื่อจำกัดจำนวนเสื้อผ้าต่อบุคคล โดยเลือกเสื้อผ้าส่วนบนได้ไม่เกิน 1 รายการ และส่วนล่างได้ไม่เกิน 1 รายการ ในกรณีที่มี dress จะพิจารณาเป็น full body item

ตารางที่ 4.8 แสดงตัวอย่างผลทำนายและผลที่ระบบเลือก

| ลำดับผลทำนาย | Class | Confidence | การตัดสินใจ |
|---:|---|---:|---|
| 1 | `long_sleeve` | 0.93 | เลือกเป็นเสื้อผ้าส่วนบน |
| 2 | `trousers` | 0.88 | เลือกเป็นเสื้อผ้าส่วนล่าง |
| 3 | `shorts` | 0.22 | ไม่เลือก เพราะมีส่วนล่างแล้ว |

การใช้กฎนี้ช่วยให้ผลลัพธ์ของระบบมีความกระชับและสอดคล้องกับการค้นหาของผู้ใช้ เช่น ผู้ใช้มักค้นหาจากเสื้อและกางเกง/กระโปรงเป็นหลัก ไม่ได้ต้องการผลทำนายทุก class ที่ classifier ให้มา การเลือกผลลัพธ์แบบนี้จึงช่วยลด noise ของข้อมูลและทำให้ query ในขั้นตอนค้นหาทำงานชัดเจนขึ้น

## 4.4 ผลการทดสอบการจัดเก็บข้อมูล

การทดสอบ database payload มีเป้าหมายเพื่อตรวจสอบว่าข้อมูลที่ `VideoProcessor` เตรียมก่อนส่งเข้า `DatabaseService.insert_detections_batch` มี field สำคัญครบถ้วนหรือไม่ โดยไม่ได้เชื่อมต่อ PostgreSQL จริง แต่ใช้ fake database เพื่อรับ payload ที่ระบบสร้างขึ้น การทดสอบลักษณะนี้ช่วยตรวจสอบ schema contract ระหว่าง service layer และ database layer ได้โดยไม่ต้องขึ้นกับสถานะของฐานข้อมูลจริง

ผลการทดสอบพบว่าระบบสร้าง payload ได้ครบตาม contract ที่จำเป็นต่อการบันทึกผลการตรวจจับ ได้แก่ `camera_id`, `track_id`, `category`, `class_name`, `bbox`, `image_path`, `video_time_offset`, `video_id` และ `embedding` โดย embedding มีขนาด 768 มิติ และจำนวนรายการที่บันทึกใน fake database เท่ากับ 1 รายการ

การทดสอบ advanced search มีเป้าหมายเพื่อตรวจสอบว่าเมื่อผู้ใช้กำหนดเงื่อนไขค้นหา ระบบสามารถสร้าง SQL WHERE clause และ parameter ได้ตรงกับเงื่อนไขหรือไม่ ตัวอย่างเงื่อนไขที่ใช้ทดสอบคือค้นหาในกล้อง `CH4-CAM-01` วิดีโอ `chapter4-video-flow` โดยต้องการคนที่มี `long_sleeve` สี `navy_blue` หรือ `black` และมี `trousers` สีใดก็ได้ ภายใต้ global logic แบบ AND

## 4.6 ปัญหาที่พบและการแก้ไข

ระหว่างการสร้างชุดทดสอบใหม่พบปัญหาหนึ่งใน flow การอ่านวิดีโอแบบ direct read เมื่อใช้ `frame_skip` เท่ากับ 4 กับวิดีโอ 24 เฟรม ระบบควรประมวลผล 6 เฟรม แต่ก่อนแก้ไขระบบประมวลผล 8 เฟรม เนื่องจากตัวแปร `frame_number` ถูกเพิ่มค่าทั้งก่อนประมวลผลและท้ายลูป ทำให้การนับเฟรมคลาดเคลื่อนจากค่าที่ควรเป็น

หลังจากแก้ไข logic โดยไม่เพิ่ม `frame_number` ซ้ำก่อนการประมวลผล ผลลัพธ์กลับมาตรงกับ expected behavior คือประมวลผลเฟรม 0, 4, 8, 12, 16 และ 20 รวม 6 เฟรม นอกจากนี้ยังมีการเพิ่ม metric ที่จำเป็นต่อการเขียนบทที่ 4 ได้แก่ `total_detections`, `unique_persons`, `duration_seconds`, `effective_fps` และ `completed`

ตารางที่ 4.16 ปัญหาและแนวทางแก้ไขที่พบจากชุดทดสอบใหม่

| ปัญหา | ผลกระทบ | แนวทางแก้ไข | สถานะ |
|---|---|---|---|
| การนับ frame skip ใน direct video flow ไม่ตรง | จำนวนเฟรมที่ประมวลผลมากกว่าที่ควรเป็น | ลบการเพิ่มค่า `frame_number` ที่ซ้ำก่อนประมวลผล | แก้ไขแล้ว |
| metric `total_detections` ไม่ถูกนับครบ | บทที่ 4 ไม่มีตัวเลข detection ที่เชื่อถือได้จาก stats | เพิ่มการนับจำนวน detection จาก `result.detections` | แก้ไขแล้ว |
| metric `unique_persons` ไม่มีค่า | ไม่สามารถสรุปจำนวนบุคคลไม่ซ้ำได้ | เพิ่ม set สำหรับเก็บ `persistent_id` หรือ `track_id` | แก้ไขแล้ว |
| `duration_seconds` และ `effective_fps` ไม่มีค่าที่ใช้ได้ | ไม่สามารถรายงานประสิทธิภาพเบื้องต้นได้ | ตั้งค่า `start_time` และ `end_time` ใน stats | แก้ไขแล้ว |

การพบปัญหานี้สะท้อนให้เห็นว่าชุดทดสอบใหม่มีประโยชน์ต่อการตรวจสอบความถูกต้องของ flow ภายในระบบ เพราะสามารถจับปัญหาที่ส่งผลต่อข้อมูลบทที่ 4 ได้ก่อนนำผลลัพธ์ไปเขียนรายงาน หากไม่มีชุดทดสอบลักษณะนี้ จำนวนเฟรมที่ประมวลผลอาจถูกนำไปสรุปผิดในบทผลการทดลองได้

## 4.7 การวิเคราะห์ผลตามวัตถุประสงค์

| วัตถุประสงค์ | ตัวชี้วัดความสำเร็จ | ผลการประเมิน |
|---|---|---|
| พัฒนาระบบช่วยค้นหาบุคคลจากวิดีโอกล้องวงจรปิด | มี flow รับวิดีโอ ประมวลผล และส่งผลให้ค้นหาได้ | สำเร็จระดับต้นแบบ |
| ตรวจจับบุคคลและลักษณะเสื้อผ้า | FrameProcessor ส่งต่อข้อมูลคน เสื้อผ้า สี และ embedding ได้ครบ | สำเร็จระดับ flow/contract |
| จัดเก็บข้อมูลผลตรวจจับ | payload มี field สำคัญสำหรับฐานข้อมูลครบ | สำเร็จระดับ contract |
| พัฒนาเงื่อนไขค้นหาจากเสื้อผ้าและสี | advanced search สร้าง query จากเงื่อนไขผู้ใช้ได้ | สำเร็จระดับ query contract |
| วิเคราะห์ข้อจำกัดของระบบ | ระบุข้อจำกัดด้าน model จริง, video จริง, performance และ ground truth | สำเร็จ |

จากตารางข้างต้นสามารถสรุปได้ว่าระบบผ่านการประเมินในระดับต้นแบบและระดับ contract ของ flow หลักแล้ว แต่ยังไม่สามารถสรุปผลเชิงความแม่นยำของโมเดลจริงได้ เนื่องจากยังไม่มีผลทดสอบจากวิดีโอจริงพร้อม ground truth การประเมินดังกล่าวจึงควรถูกใช้เป็นหลักฐานว่าระบบทำงานเชิงโครงสร้างได้ครบ มากกว่าจะเป็นหลักฐานด้าน accuracy ของ AI

## 4.8 การเปรียบเทียบกับระบบเดิม

ระบบเดิมในการค้นหาบุคคลจากกล้องวงจรปิดมักพึ่งพาการเปิดดูวิดีโอด้วยมนุษย์เป็นหลัก ผู้ปฏิบัติงานต้องดูวิดีโอทีละช่วงเวลา จดจำลักษณะบุคคลด้วยสายตา และเปรียบเทียบกับภาพหรือคำอธิบายเป้าหมายด้วยตนเอง วิธีดังกล่าวใช้เวลานานและมีโอกาสผิดพลาด โดยเฉพาะเมื่อวิดีโอมีหลายไฟล์ หลายกล้อง หรือมีระยะเวลานาน

ระบบที่พัฒนาขึ้นช่วยเปลี่ยนกระบวนการจากการดูวิดีโอทั้งหมดเป็นการจัดเก็บข้อมูลเชิงโครงสร้าง ทำให้สามารถค้นหาจากเงื่อนไข เช่น เสื้อผ้า สี กล้อง วิดีโอ และช่วงเวลาได้ ระบบจึงช่วยลดภาระในการคัดกรองข้อมูลเบื้องต้น แม้ยังไม่สามารถยืนยันตัวบุคคลได้สมบูรณ์หรือ track ข้ามกล้องได้จริงในขอบเขตปัจจุบัน

ตารางที่ 4.18 การเปรียบเทียบระบบเดิมกับระบบที่พัฒนา

| ประเด็นเปรียบเทียบ | ระบบเดิมแบบแมนนวล | ระบบที่พัฒนา |
|---|---|---|
| วิธีค้นหา | เปิดดูวิดีโอด้วยคน | ค้นหาจาก metadata ที่ AI วิเคราะห์ |
| เกณฑ์ค้นหา | การจดจำด้วยสายตา | ประเภทเสื้อผ้า สี กล้อง วิดีโอ และเวลา |
| ความเร็วในการคัดกรอง | ขึ้นกับผู้ปฏิบัติงาน | ลดเวลาจากการกรองข้อมูลล่วงหน้า |
| ความสม่ำเสมอ | อาจลดลงเมื่อผู้ปฏิบัติงานเหนื่อย | ใช้เงื่อนไขเดียวกันในการประมวลผล |
| การจัดเก็บหลักฐาน | มักแยกเป็นไฟล์/บันทึกด้วยมือ | มีฐานข้อมูลและ path ภาพประกอบ |
| ข้อจำกัดหลัก | ใช้แรงงานและเวลา | ยังขึ้นกับคุณภาพภาพและความแม่นยำโมเดล |
| การต่อยอด | ทำได้ยาก | รองรับการเพิ่ม search, Re-ID และ cross-camera ในอนาคต |

จากการเปรียบเทียบจะเห็นว่าระบบใหม่มีข้อได้เปรียบด้านการจัดโครงสร้างข้อมูลและการค้นหาย้อนหลัง แต่ยังมีข้อจำกัดที่ต้องพิจารณา เช่น ความถูกต้องของโมเดล ความไวต่อแสงและสี ความละเอียดของภาพ และการติดตามข้ามกล้องที่ยังอยู่นอกขอบเขตปัจจุบัน

## 4.9 การอภิปรายผล

ผลการทดสอบใหม่แสดงให้เห็นว่าระบบมีโครงสร้าง flow ที่สามารถตรวจสอบได้เป็นลำดับ ตั้งแต่การประมวลผลภาพหนึ่งเฟรมไปจนถึงการเตรียมข้อมูลสำหรับค้นหาและบันทึกผล การแยกทดสอบแต่ละส่วนช่วยให้ระบุปัญหาของระบบได้ชัดเจนกว่าการทดสอบจากผลลัพธ์สุดท้ายเพียงอย่างเดียว ตัวอย่างเช่นปัญหา frame skip ที่ทำให้จำนวนเฟรมประมวลผลไม่ตรงกับที่คาดไว้ถูกตรวจพบและแก้ไขได้จากชุดทดสอบนี้

อีกประเด็นสำคัญคือ embedding dimension ที่ได้จาก flow ปัจจุบันมีขนาด 768 มิติ ซึ่งควรใช้ให้สอดคล้องกับบทที่ 2 และบทที่ 3 หากเอกสารส่วนใดระบุว่า feature vector มีขนาด 512 มิติ ควรปรับให้ตรงกับ implementation จริง หรือระบุให้ชัดเจนว่าเป็นแนวคิดเดิมก่อนปรับระบบ ประเด็นนี้สำคัญเพราะขนาด vector ส่งผลต่อทั้งฐานข้อมูล การค้นหา similarity และคำอธิบายเชิงเทคนิคของระบบ

สำหรับผลด้านประสิทธิภาพ เช่น `effective_fps` ที่รายงานจากชุดทดสอบนี้ ไม่ควรนำไปสรุปว่าเป็น FPS จริงของระบบบน GTX 1650 เนื่องจาก test ใช้ fake AI และ synthetic video ทำให้เวลาในการประมวลผลต่ำกว่าการใช้โมเดลจริงมาก ค่าดังกล่าวใช้เพื่อยืนยันว่า field ใน stats ทำงานได้เท่านั้น ไม่ใช่ตัวเลข performance ขั้นสุดท้าย

ด้านการค้นหา ผล query contract แสดงว่าระบบสามารถสร้างเงื่อนไข SQL จากข้อมูลเสื้อผ้าและสีได้ แต่ยังไม่ได้ยืนยันว่าผลค้นหาที่ได้ตรงกับคนเป้าหมายมากน้อยเพียงใด การประเมินคุณภาพการค้นหาจึงควรมีชุด query พร้อมคำตอบจริง เช่น กำหนดว่าในวิดีโอหนึ่งมีคนใส่เสื้อแขนยาวสีน้ำเงินปรากฏช่วงเวลาใด แล้ววัดว่าระบบค้นหาเจอในอันดับต้น ๆ หรือไม่

## 4.10 ข้อจำกัดของผลการทดสอบในบทนี้

แม้ว่าชุดทดสอบใหม่จะช่วยยืนยัน flow และ contract ของระบบได้ แต่ยังมีข้อจำกัดหลายประการที่ต้องระบุให้ชัด เพื่อไม่ให้ตีความผลลัพธ์เกินกว่าขอบเขตของการทดสอบ

1. ยังไม่ได้ใช้โมเดล YOLO11 และโมเดลจำแนกเสื้อผ้าจริงในการวัด accuracy
2. ยังไม่ได้ใช้วิดีโอกล้องวงจรปิดจริงที่มี ground truth
3. ยังไม่ได้วัด precision, recall, mAP, F1-score หรือ confusion matrix
4. ยังไม่ได้วัด FPS, latency, RAM และ VRAM usage บน GPU NVIDIA GTX 1650 จริง
5. ยังไม่ได้ทดสอบการเขียนและอ่านข้อมูลจาก PostgreSQL จริงในชุดทดสอบบทนี้
6. ยังไม่ได้ทดสอบคุณภาพการค้นหาด้วย query จริงเทียบกับคำตอบที่ label ไว้
7. ยังไม่ได้ทดสอบในสภาพแสงที่หลากหลาย เช่น แสงน้อย เงาเข้ม แสงสะท้อน หรือมุมกล้องสูง
8. ยังไม่ได้ทดสอบการใช้งานหลายกล้องแบบเชื่อมโยงความสัมพันธ์ระหว่างกล้อง

ตารางที่ 4.19 ข้อมูลที่ควรเพิ่มภายหลังเพื่อให้บทที่ 4 สมบูรณ์

| รายการที่ควรเพิ่ม | ข้อมูลที่ต้องมี | ใช้สรุปผล |
|---|---|---|
| ผลทดสอบวิดีโอจริง | วิดีโอจริงพร้อมจำนวนเฟรมและ resolution | จำนวน detection และตัวอย่างผลลัพธ์จริง |
| ผล accuracy ของเสื้อผ้า | test set ที่มี label 6 class | accuracy, precision, recall, F1-score |
| ผลทดสอบสี | crop เสื้อผ้าที่มี label สีจริง | color accuracy |
| ผล tracking | label identity ต่อเฟรม | ID switch และ tracking continuity |
| ผล performance | เวลา inference, RAM, VRAM | FPS และ latency บน GTX 1650 |
| ผล search quality | query พร้อมคำตอบจริง | precision@k และ recall@k |
| ภาพประกอบ | screenshot dashboard/search/detection | ภาพที่ 4.1-4.10 |

## 4.11 แนวทางการทดสอบเพิ่มเติม

เพื่อให้บทที่ 4 มีความสมบูรณ์ในเชิงวิชาการมากขึ้น ควรเพิ่มการทดสอบด้วยข้อมูลจริงและ ground truth โดยแบ่งออกเป็น 4 กลุ่ม ได้แก่ การทดสอบความแม่นยำของโมเดล การทดสอบ tracking การทดสอบ performance และการทดสอบคุณภาพการค้นหา

ตารางที่ 4.20 แผนการทดสอบเพิ่มเติมที่ควรดำเนินการ

| กลุ่มการทดสอบ | วิธีทดสอบ | ผลลัพธ์ที่ควรได้ |
|---|---|---|
| Detection accuracy | ใช้วิดีโอหรือภาพที่ label bbox บุคคล | precision, recall, mAP |
| Clothing classification | ใช้ test set 6 class ที่มี label จริง | accuracy, F1-score, confusion matrix |
| Color accuracy | ใช้ crop เสื้อผ้าที่ label สีหลัก | color accuracy และ error cases |
| Tracking continuity | ใช้วิดีโอที่ label identity ต่อเฟรม | ID switch, track continuity |
| Performance | รันโมเดลจริงบน GTX 1650 | FPS, latency, RAM, VRAM |
| Search quality | สร้าง query พร้อม expected result | precision@k, recall@k |
| Usability | ให้ผู้ใช้ทดลองค้นหา | เวลาเฉลี่ยในการค้นหาและ feedback |

หากมีเวลาและข้อมูลเพียงพอ ควรเลือกอย่างน้อย 2 กลุ่มที่สำคัญที่สุดคือ performance บนเครื่องจริงและ accuracy ของเสื้อผ้า เพราะเป็นตัวเลขที่กรรมการมักใช้พิจารณาว่าระบบใช้งานได้จริงมากน้อยเพียงใด

## 4.12 ข้อเสนอแนะจากผลการทดสอบ

จากผลการทดสอบและข้อจำกัดที่พบ สามารถสรุปข้อเสนอแนะในการปรับปรุงระบบได้ดังนี้

1. เพิ่ม integration test ที่ใช้ PostgreSQL จริง เพื่อยืนยันว่าข้อมูลจาก payload ถูกบันทึกและค้นหาได้ครบ
2. เพิ่ม test ที่รันโมเดลจริงกับวิดีโอสั้น เพื่อให้ได้ตัวเลข processing time และ FPS จริง
3. ทำ test set ขนาดเล็กสำหรับเสื้อผ้า 6 class เพื่อคำนวณ accuracy และ confusion matrix
4. ทำชุดภาพ crop สำหรับทดสอบสี โดยเลือกสภาพแสงปกติ แสงน้อย และมีเงา เพื่อวิเคราะห์ข้อจำกัดของระบบสี
5. เพิ่ม screenshot ของหน้าจอหลักในบทที่ 4 เพื่อให้ผลการพัฒนาระบบมีหลักฐานเชิงภาพ
6. ปรับเอกสารบทที่ 2 และบทที่ 3 ให้ตรงกับ embedding 768 มิติและสถานะการใช้ PostgreSQL/JSONB จริง
7. เพิ่ม logging สำหรับเวลา inference ของแต่ละขั้นตอน เช่น detection, classification, color analysis และ database insert

ข้อเสนอแนะเหล่านี้จะช่วยให้บทที่ 4 มีน้ำหนักมากขึ้น เพราะไม่ได้แสดงเพียงว่าระบบทำงานได้ แต่ยังแสดงว่าผู้พัฒนารู้ข้อจำกัดของระบบและมีแนวทางทดสอบหรือปรับปรุงต่ออย่างเป็นระบบ

## 4.13 สรุปบท

บทนี้ได้นำเสนอผลการดำเนินงานและผลการทดสอบระบบโดยใช้ชุดทดสอบใหม่ที่ออกแบบจาก flow การทำงานจริงของระบบ ผลการทดสอบผ่านทั้งหมด 5 รายการ โดยยืนยันได้ว่าระบบสามารถประมวลผลข้อมูลตาม contract สำคัญ ได้แก่ การประมวลผลภาพแบบสองรอบ การประมวลผลวิดีโอตาม frame skip การเตรียม payload สำหรับฐานข้อมูล และการสร้าง query สำหรับ advanced search

ผลการทดสอบยังช่วยค้นพบและแก้ไขปัญหาใน `VideoProcessor` ได้แก่การนับ frame skip ที่คลาดเคลื่อน และเพิ่ม metric ที่จำเป็นต่อการเขียนบทที่ 4 เช่น `total_detections`, `unique_persons`, `duration_seconds` และ `effective_fps` อย่างไรก็ตามผลในบทนี้เป็นการทดสอบ flow และ data contract จึงยังต้องเพิ่มผลจากโมเดลจริง วิดีโอจริง และ ground truth ภายหลัง เพื่อให้สามารถสรุปความแม่นยำและประสิทธิภาพของระบบได้อย่างสมบูรณ์

โดยสรุป ระบบที่พัฒนาขึ้นมีความพร้อมในระดับต้นแบบและมีโครงสร้างที่รองรับการค้นหาบุคคลจากกล้องวงจรปิดด้วยข้อมูลเสื้อผ้า สี และ metadata ของวิดีโอ แต่ยังต้องผ่านการทดสอบเชิงภาคสนามและเชิงสถิติเพิ่มเติมก่อนสรุปว่าเหมาะสมต่อการใช้งานจริงในทุกสถานการณ์
